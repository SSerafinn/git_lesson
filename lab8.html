Changes, not Files

1

Goals

Understand that Git works with the changes, not the files themselves.

Unlike most version control systems that track files, Git tracks changes. A git add file command does not permanently add the file, but rather notes the current state of the file so that it can be committed later.

2

First Change: Adding default page tags

Update hello.html with HTML and BODY tags:

File: hello.html
< html>
  < body>
    < h1>Hello, World!< /h1>
  < /body>
< /html>
3

Add this change

Run
git add hello.html
4

Second Change: Add the HTML headers

Now add a HEAD section:

File: hello.html
< html>
  < head>< /head>
  < body>
    < h1>Hello, World!< /h1>
  < /body>
< /html>
5

Check the current status

Run
git status
Result
$ git status
On branch main
Changes to be committed:
  modified: hello.html

Changes not staged for commit:
  modified: hello.html

Notice that hello.html is listed twice: - The first change (default tags) is staged and ready. - The second change (HEAD section) is unstaged.

6

Commit

Run
git commit -m "Added standard HTML page tags"
git status
Result
[main 46afaff] Added standard HTML page tags
 1 file changed, 5 insertions(+), 1 deletion(-)
$ git status
On branch main
Changes not staged for commit:
  modified: hello.html
7

Adding the second change

Stage the second change:

Run
git add .
git status
Result
$ git add .
$ git status
Changes to be committed:
  modified: hello.html
8

Commit the second change

Run
git commit -m "Added HTML header"
Result
[main b7614c1] Added HTML header
 1 file changed, 2 insertions(+)
